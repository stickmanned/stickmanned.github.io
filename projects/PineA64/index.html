Pine A64 Gaming PC

Built for Hack Club Horizons • Started Jun 2025 • Last updated 2026-07-02

This page documents how I turned a Pine A64 single-board computer into a fully functional, custom-cooled, custom-enclosed cloud/retro gaming PC. The board runs DietPi and streams games from a host gaming PC over Moonlight using hardware-accelerated video decode. Active cooling (a Noctua fan on a boosted 12V rail) keeps it stable under sustained streaming load, and the whole thing lives in a 3D-printed enclosure I designed with a snap-fit power button. I built it for Hack Club Horizons.

Goal of the build: take a cheap, low-power ARM64 board that was never meant for gaming and get it to play modern titles like Roblox and BeamNG.drive smoothly — by offloading the rendering to a host PC and using the Pine A64 purely as a low-latency Moonlight streaming client, kept cool and quiet inside a case I made myself.

Overview

The Pine A64 is a 2GB ARM64 board that costs almost nothing and draws almost no power — but it has no real GPU to speak of, so it can't render a modern game on its own. Instead of fighting that, I leaned into it: the board does zero game rendering. A host gaming PC does all the heavy lifting, and the Pine A64 just decodes and displays the video stream over Moonlight, sending my inputs back. That turns a tiny SBC into a capable "gaming PC" as long as it can decode 1080p video and stay cool doing it.

Getting there meant solving three problems: keeping the board cool without a huge cooler, getting the Allwinner A64 graphics stack to do hardware-accelerated decode under Linux, and packaging it all into an enclosure that looks and feels like a real product.

Wiring & Electrical

The most important rule of this build is that there are two separate voltage rails that must never touch each other: 5V powers the Pine A64, and 12V powers only the Noctua fan. The MT3608 boost converter is the only thing that ever sees 12V — everything on the board side stays at 5V.

5 V PSU
  ├── PINE A64 power input (microUSB, or Euler pin 2/4 → GND pin 6/9)
  └── MT3608 boost converter input (IN+ / IN-)

MT3608 output, adjusted to 12.0 V
  └── Noctua fan (red = OUT+, black = OUT-, yellow = disconnected)

Key switch (momentary, normally-open)
  └── EXP header pin 5 (Pwr/Stb Sw) ↔ EXP header pin 6 (GND)

I split the 5V rail so the fan's boost converter draws directly from the PSU rather than through the board, which keeps the fan's startup current off the Pine A64. Before ever connecting the fan, I measured the MT3608 output with a multimeter and adjusted its potentiometer to exactly 12.0V. The fan's yellow tachometer wire isn't used here, so it stays disconnected and insulated.

The power button is wired to the Pine A64's dedicated EXP header — pins 5 (Pwr/Stb Sw) and 6 (GND) — not in series with the main power line. That way the switch signals the board to power on or shut down cleanly instead of hard-cutting 5V, which would risk corrupting the microSD card if the OS is running.

There's no custom PCB — the electronics are point-to-point wiring around off-the-shelf parts, so the reproducible hardware artifact is the exported wiring schematic PDF.

Software

The board runs DietPi for Allwinner A64, flashed to a microSD card. The interesting part was getting hardware-accelerated video decode working, which is what makes streaming smooth instead of a slideshow.

The graphics fix

Out of the box, the Allwinner A64 DRM/KMS stack would exhaust its CMA memory pool trying to bring up the display, so I force HDMI to 1080p before Linux starts by adding video=HDMI-A-1:1920x1080@60 to the DietPi boot environment. I also purged the XFCE desktop and LightDM entirely so Moonlight can run directly on the DRM/KMS path instead of going through a windowing system, and added a udev rule so the normal login can access /dev/dri/* and /dev/input/*.

Streaming

Moonlight (moonlight-qt) is installed on the board and paired with the host gaming PC. An autostart block in the DietPi user's shell profile launches straight into a Moonlight session on boot, so powering the device on drops you right into streaming with no desktop in between. All of these config files are committed as real files in firmware-config/ so the setup is reviewable and reproducible.

Enclosure & 3D Printing

I designed the case in Fusion 360 — a top and bottom shell, a fan mount, and a power-button cutout. The cutout ended up slightly undersized for a standard keycap, so I generated a correctly-sized cap with the power icon engraved on it using KeyV2 in OpenSCAD.

The two shells were printed on two different printers, each tuned for what that part needs — surface finish on the top, load-bearing strength on the bottom:

Build Story

I built this over a roughly 24-hour session for Hack Club Horizons, and I kept an hourly devlog the whole way through. The build broke down into teardown, adding cooling, bringing up the OS, the DRM/KMS graphics debugging saga, designing the enclosure in CAD, final wiring, and the custom OpenSCAD keycap.

The hardest part by far was the graphics stack. For several hours the board just would not do hardware-accelerated decode — it kept running out of contiguous memory during display init. Narrowing that down to the CMA pool and the boot-time HDMI mode, and then chasing a few boot-time race conditions on top of it, took the most patience of anything in the build. Once the DRM/KMS path finally lit up and modetest recognized the Allwinner display engine, the rest came together quickly.

Along the way I hit the kind of small physical problems you only find by actually building: hot-plugging USB peripherals caused brownout resets, so peripherals have to be cold-plugged before boot; a fan-mount pillar blocked the fan and had to be redesigned out; and cable management inside a tiny shell meant hot-gluing the fan leads and MT3608 down for strain relief. The full hour-by-hour writeup, including root-cause notes on each of these, is in the Devlog.

Safety Notes

  • All wiring is low-voltage DC (5V/12V) — no mains voltage anywhere in the enclosure.
  • Never connect the MT3608's 12V output to the Pine A64. 12V is only for the fan; the board only ever sees 5V.
  • Always verify the MT3608 output at 12.0V with a multimeter before connecting the fan.
  • Wire the power switch across the EXP header's Pwr/Stb Sw and GND pins only — never in series with the main 5V line, which can corrupt the microSD card.
  • No lithium batteries are used in this build.
